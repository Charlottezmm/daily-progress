# PawPlan v1 Formal Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship PawPlan v1 formal for stable personal daily use and invite-based sharing, with reliable high-level agent rebalance tools, observable agent runs, and hard verification gates.

**Architecture:** Keep scheduling outside PawPlan and keep Review as the only apply boundary. Add an `agent_runs` reliability layer plus high-level MCP rebalance tools that convert agent intent into existing Review drafts through the current patch validation path.

**Tech Stack:** Next.js 14, TypeScript, Drizzle ORM, Postgres, Zod, Vitest, Playwright, hosted MCP.

---

## Scope

This plan implements the reliability core for v1 formal. It does not add app-owned AI, built-in scheduler, auto-apply, billing, teams, public open signup, or calendar sync.

Use the technical design as the source of truth:

```text
docs/superpowers/specs/2026-06-17-pawplan-v1-formal-technical-design.md
```

```ts
  inputJson: jsonb("input_json").notNull(),

  errorJson: jsonb("error_json"),

  createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
  updatedAt: timestamp("updated_at", { withTimezone: true }).notNull().defaultNow(),
}, (table) => ({
  uniqueWorkspaceIdempotency: uniqueIndex("agent_runs_workspace_idempotency_unique").on(
    table.workspaceId,
    table.idempotencyKey,
  ),
  workspaceCreatedIdx: index("agent_runs_workspace_created_idx").on(table.workspaceId, table.createdAt),
  workspaceStatusIdx: index("agent_runs_workspace_status_idx").on(table.workspaceId, table.status),
}));
```

## Long-Term Development Reliability Rules

Carry these rules into future PawPlan and other projects:

- Start with baseline verification: `git status`, focused tests, and existing docs.
- Define a verifiable completion standard before implementation.
- Prefer narrow product APIs over asking external agents to assemble strict low-level schemas.
- For unattended agent workflows, every run needs a persisted status, idempotency key, and readback.
- Do not treat tool output as success unless the returned body proves the intended state.
- Production fixes need smoke plus data readback, not only logs.
- UI changes need regression checks on the workflow they affect.
- Large work should be split across subagents with review after each bounded task.
- Do not expand v1 into scheduler, app-owned AI, billing, teams, or calendar sync.
