### 1.3 关键流程

#### 1.3.1 首次启动（迁移流程）
```
启动 → /api/plan GET → KV 没数据
     → 显示 onboarding-or-skip 提示
     → "用 5 月数据开箱即用" 按钮 → 调 /api/plan PUT 写入预置 plan
     → 跳 /today
```

#### 1.3.2 Onboarding（导入文档/文件拆解）
```
1. 用户访问 /onboarding
2. 提供 3 种输入方式（互斥）：
   a. 直接粘文字到 textarea
   b. 上传文件 (.md / .txt / .pdf) → POST /api/plan/parse-file
      → 服务端返回提取的文本 → 填入 textarea 让用户预览/编辑
      （PDF 提取常乱码，强制让用户先看再交 AI）
   c. 上传 plan-template.json → 跳过 AI 直接到编辑视图（模板分享路径）
3. 额外输入：
   - totalDays (默认 30)
   - startDate (默认今天)
   - 可选：勾选"使用现有 Goal 作为约束"（如已存 goal:current）
4. 点"AI 拆解" → POST /api/plan/generate (stream)
   - 服务端 prompt 自动拼接：sourceText + goal (如启用) + inheritUnfinished (如启用)
5. SSE 流式输出 JSON (用户能看到 weeks/days 一格一格出现)
6. 完成后进入"编辑视图" — 每周/每天/每个 task 可改 (含 tags / priority)
7. 点"保存为 baseline" → PUT /api/plan (带 confirmOverwrite=true)
8. 如果 onboarding 时还没 goal，提示用户写一段 goal → PUT /api/goal
9. 跳 /today
```

#### 1.3.3 每日打开 /today
```
1. RSC 加载 plan + snapshot:{today}
2. snapshot 不存在 → 从 baseline (plan.days[i].defaultTasks) 生成默认 snapshot 写入
3. 渲染：Timeline + TaskRow[] + ChatDrawer 折叠
4. 用户编辑任务 → 客户端立即更新 + debounced PUT /api/snapshot/[date]
5. 用户点"AI 重规划今天" → POST /api/regenerate (stream) → AI 输出新 snapshot
   → 用户审 → 接受/拒绝
6. 用户点"AI 协作"展开 chat → 输入 → POST /api/chat (stream)
   → AI 普通回复 OR AI 返回 patch (tool call) → 应用到 snapshot
```

#### 1.3.4 复盘（昨晚或今晨）
```
1. 用户在 /today 顶部"昨日复盘"卡（如果 reflection:{yesterday} 不存在）
2. 写一段文字 → PUT /api/reflection/[yesterday]
3. 完成后弹 toast "AI 重规划今天？" → 是 → 调用 1.3.3 的重规划
```

#### 1.3.5 月度重生成（覆盖 plan + 自动归档 + 带未完成）
```
触发点：
- /today 顶部 banner（plan 剩 ≤ 3 天 / 已过期）
- 设置页"重新规划" 按钮
- 6 月 1 日打开发现 plan.endDate < today → 强制跳

流程：
1. 跳 /onboarding（参数 mode=continue）
2. 服务端在 GET /onboarding-context 时计算：
   a. 现有 goal:current 内容
   b. 上月 plan archive 的最后 7 天 snapshot
   c. 跑一次轻量 AI 分析：列出"上月未完成 / 部分完成"任务（最多 5 条）
3. /onboarding 默认勾选"使用 Goal" + "带入未完成（5 条建议）"
4. 用户可逐条勾选要不要带、可编辑文字
5. 用户输入 sourceText（描述本月重点）
6. POST /api/plan/generate (stream)：
   - prompt 包含 goal + sourceText + 勾选的未完成
   - AI 把未完成项优先排到第 1 周
7. AI 输出 → 编辑视图 → PUT /api/plan：
   a. 复制旧 plan:current 到 plan:archive:{ISO_TS}
   b. 写入新 plan:current
   c. snapshot:{date} / reflection:{date} 不动（历史保留）
   d. goal:current 不动
8. 跳 /today
```

#### 1.3.6 模板分享（导出 / 导入 plan baseline）
```
导出：设置页"导出当前 plan" → 下载 plan-{title}-{ts}.json (只含 plan，不含 snapshot)
导入：/onboarding 上方"从 JSON 导入" → 选文件 → Zod 校验 → 显示在编辑视图 → 保存走 1.3.5 流程
```

#### 1.3.7 Goal 管理（长期目标，独立于 plan）
```
设置页 → "我的长期目标"
- 一段 markdown 文字（≤ 5000 字）+ 几条 long-term milestones（≤ 10 条）
- 例：「4 个月学 LLM 内功 + Agent 工程；月底前发出 v0.1 开源 Agent 工具」
- 月度重生成 plan 时 AI 把 goal 作为硬约束（不能写"今天去旅游"这种偏离）
- GET/PUT /api/goal
- Goal 变更不影响当前 plan，但下次重生成会反映新 goal
```

### 1.4 技术选型 + 理由

| 选型 | 理由 |
|---|---|
| **Next.js 15 App Router** | RSC 让数据加载零客户端 JS、API route 跟前端同 repo、Vercel 一键部署、生态最强 |
| **TypeScript** | 数据模型严格、避免 baseline 跟 snapshot schema 漂移 |
| **Tailwind v4** | 跟 shadcn 配套；CSS 变量调主题方便 |
| **shadcn/ui** | 不是 npm 包，是 cli 拷贝代码到本地 → 完全可控、改 UI 不用 fight 库的默认值 |
| **Zustand** | 客户端 UI 状态（drawer 开关、当前选中日）极简 store；没必要 Redux |
| **TanStack Query** | 缓存 server data + optimistic update + revalidate；自带的 mutation 比手写 fetch 干净 |
| **Vercel KV** | 部署在 Vercel 直接 1-click setup；Upstash Redis 免费档 30K 命令/月，单人用永远不超 |
| **OpenAI SDK** | DeepSeek 官方文档示例就用它；社区最熟悉；流式开箱即用 |
| **Zod** | runtime schema 校验 + 自动生成 TS 类型；server/client 共享 schema |
| **date-fns** | tree-shakable、纯函数；moment 重且 deprecated |
| **react-markdown** | 渲染 chat 里的 markdown / 复盘 |

**故意不选：**
- ❌ Prisma + Postgres：单人小数据，KV 足够；引入 ORM 是过度工程
- ❌ tRPC：API 简单，REST + zod 够，tRPC 学习曲线和单人 ROI 不匹配
- ❌ Redux / Jotai：Zustand 已经够轻
- ❌ Framer Motion（暂不引入）：v1 用 CSS transition 够；如果 v2 要做"AI 改任务时屏幕动效"再加
- ❌ Auth.js：单用户，secret token + middleware 足够

### 1.5 非功能性需求

| 维度 | 目标 |
|---|---|
| 性能 | TTFB < 200ms（KV 同区域）、AI 流式首 token < 2s |
| 可用性 | Vercel SLA，单人用不用考虑高可用 |
| 安全 | API key 服务端 only；所有 API 加 secret token 防爬虫；KV 数据不含敏感信息 |
| 成本 | DeepSeek ¥3-10/月；Vercel KV 免费档；Vercel hosting 免费档（Hobby） |
| 可维护性 | TypeScript 严格模式；ESLint + Prettier；commit 走 Husky + lint-staged |
| 可扩展 | 数据 schema 留 `version` 字段；plan 数据用 stable id 不用 index；多 plan 升级只需加 `plan:{id}` 维度 |

## 2. 数据模型定义（TypeScript + Zod）

### 2.1 核心 schemas (`lib/schemas.ts`)

```typescript
import { z } from "zod";

// ===== 时间 =====
export const TimeBlockSchema = z.object({
  start: z.string().regex(/^\d{2}:\d{2}$/),  // "05:00"
  end:   z.string().regex(/^\d{2}:\d{2}$/),  // "07:00"
});
export type TimeBlock = z.infer<typeof TimeBlockSchema>;

// ===== 任务种类 =====
export const TaskKindSchema = z.enum(["学", "项", "媒", "休"]);
export type TaskKind = z.infer<typeof TaskKindSchema>;

// ===== 优先级 =====
export const PrioritySchema = z.enum(["low", "normal", "high", "urgent"]);
export type Priority = z.infer<typeof PrioritySchema>;
// UI 颜色映射：low=#9b9b9f / normal=#6e6e73 / high=#c2410c / urgent=#dc2626

// ===== 单个任务 =====
export const TaskSchema = z.object({
  id: z.string(),                      // uuid，稳定标识
  kind: TaskKindSchema,
  blocks: z.array(TimeBlockSchema).min(1),  // 一个任务可能多时段
  content: z.string().min(1),
  done: z.boolean().default(false),
  source: z.enum(["baseline", "ai", "user"]).default("baseline"),
  // ↓ v1 §0 第 18 条新增
  tags: z.array(z.string()).default([]),    // 自由约定，如 ["micrograd", "#proj-X", "重点"]
  priority: PrioritySchema.default("normal"),
  // ↓ §0 第 19 条新增：跨月延续追溯
  inheritedFrom: z.string().nullable().default(null), // 如 "plan:archive:2026-05-31T..."
});
export type Task = z.infer<typeof TaskSchema>;

// ===== 单天 baseline (在 plan 里) =====
export const PlanDaySchema = z.object({
  num: z.number().int().min(1),
  date: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),  // "2026-05-07"
  weekday: z.string(),                            // "周四"
  weekNum: z.number().int().min(1),
  type: z.enum(["work", "rest"]),
  title: z.string(),
  defaultTasks: z.array(TaskSchema),
  selfCheck: z.string(),                          // 自检问题
});
export type PlanDay = z.infer<typeof PlanDaySchema>;

// ===== 单周 (在 plan 里) =====
export const PlanWeekSchema = z.object({
  num: z.number().int().min(1),
  dateRange: z.string(),
  theme: z.string(),
  note: z.string(),
  milestones: z.array(z.object({
    id: z.string(),
    text: z.string(),
    done: z.boolean().default(false),
  })),
});
export type PlanWeek = z.infer<typeof PlanWeekSchema>;

// ===== 整个 plan (baseline) =====
export const PlanSchema = z.object({
  version: z.literal(1),
  meta: z.object({
    title: z.string(),
    mainTheme: z.string(),
    sourceText: z.string(),                       // 用户原始输入文档
    startDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
    totalDays: z.number().int().min(1),
    createdAt: z.string(),                        // ISO
    updatedAt: z.string(),
  }),
  weeks: z.array(PlanWeekSchema),
  days: z.array(PlanDaySchema),
});
export type Plan = z.infer<typeof PlanSchema>;

// ===== 每日 snapshot (实际计划，覆盖 baseline) =====
export const SnapshotSchema = z.object({
  version: z.literal(1),
  date: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  title: z.string(),
  selfCheck: z.string(),
  tasks: z.array(TaskSchema),
  updatedAt: z.string(),
  regeneratedAt: z.string().optional(),           // 上次 AI 重规划时间
});
export type Snapshot = z.infer<typeof SnapshotSchema>;

// ===== 复盘 =====
export const ReflectionSchema = z.object({
  version: z.literal(1),
  date: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),  // 复盘的是哪天
  text: z.string(),                               // 自由文本
  createdAt: z.string(),
});
export type Reflection = z.infer<typeof ReflectionSchema>;

// ===== Chat 消息 =====
export const ChatMessageSchema = z.object({
  id: z.string(),
  role: z.enum(["user", "assistant", "system"]),
  content: z.string(),
  ts: z.string(),
  // 如果 AI 返回了 patch，记录下来便于审计
  appliedPatch: z.array(z.object({
    op: z.enum(["add", "remove", "update"]),
    taskId: z.string().optional(),
    task: TaskSchema.optional(),
  })).optional(),
});
export type ChatMessage = z.infer<typeof ChatMessageSchema>;

export const ChatHistorySchema = z.object({
  version: z.literal(1),
  date: z.string(),
  messages: z.array(ChatMessageSchema),
});
export type ChatHistory = z.infer<typeof ChatHistorySchema>;

// ===== Goal (长期目标，独立于 plan) =====
export const GoalSchema = z.object({
  version: z.literal(1),
  text: z.string().max(5000),                       // markdown，长期目标描述
  longTermMilestones: z.array(z.object({
    id: z.string(),
    text: z.string(),
    targetDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).nullable().default(null),
    done: z.boolean().default(false),
  })).max(10),
  updatedAt: z.string(),
});
export type Goal = z.infer<typeof GoalSchema>;

// ===== Settings =====
export const SettingsSchema = z.object({
  version: z.literal(1),
  thinkingForRegenerate: z.boolean().default(true),
  thinkingForChat: z.boolean().default(false),
  defaultModel: z.string().default("deepseek-v4-pro"),
});
export type Settings = z.infer<typeof SettingsSchema>;
```

### 2.2 KV Key 设计

| Key | Value Type | 备注 |
|---|---|---|
| `goal:current` | `Goal` | 长期目标 (4 个月主线)，跨 plan 不变 |
| `plan:current` | `Plan` | 当前激活的 baseline，单 plan 只一份 |
| `plan:archive:{ISO_TS}` | `Plan` | 月度覆盖时自动备份（只读，无清理逻辑，KV 数据极小不需要清） |
| `snapshot:{YYYY-MM-DD}` | `Snapshot` | 每日实际计划 |
| `reflection:{YYYY-MM-DD}` | `Reflection` | 每日复盘 |
| `chat:{YYYY-MM-DD}` | `ChatHistory` | 每日聊天，TTL 7 天（自动清） |
| `settings` | `Settings` | 全局设置 |
| `_meta:initialized` | `boolean` | 是否初始化过 |

**KV TTL 策略**：chat 设 7 天 TTL（避免数据膨胀），其他无 TTL（用户手动清）。

### 2.3 数据流向规则

1. **Plan 是只读的 baseline**：除 onboarding / "重新生成 plan" 之外不修改
2. **Snapshot 是当天的"事实"**：所有编辑、勾选、AI 改动都写 snapshot，不动 plan
3. **Snapshot 不存在则从 baseline 生成**：`getOrCreateSnapshot(date)` 是核心读取逻辑
4. **Task.id 是 uuid 而非 index**：保证 patch 操作可寻址（"remove task xyz"）

## 3. 接口契约（REST API）

### 通用约定

- 所有 endpoint 都在 `/api/*` 下
- Auth：每个 endpoint 检查 `Cookie: app_token=<APP_SECRET>`，不匹配返 401
- Content-Type：`application/json`，流式 endpoint 用 `text/event-stream` (SSE)
- 错误：`{ error: { code: string, message: string, details?: any } }` HTTP 4xx/5xx
- 时间戳：ISO 8601 UTC 字符串

### 3.1 Plan

#### `GET /api/plan`
- 200 → `Plan` 或 `null`（未初始化）

#### `PUT /api/plan`
- Body：`Plan` (Zod 校验)
- 行为：如已存在 `plan:current`，写入前先备份到 `plan:archive:{ISO_TS}` (原子操作)
- 200 → `{ ok: true, plan: Plan, archivedAs: string | null }`
- 400 → schema invalid

#### `GET /api/plan/archives`
- 200 → `Array<{ key: string, ts: string, title: string }>` — 按 ts 倒序
- 用于设置页查看历史 plan 列表（v1 只能看，不能恢复 — 恢复留 v2）

#### `POST /api/plan/generate` (stream)
- Body：
  ```ts
  {
    sourceText: string;
    totalDays: number;
    startDate: string;       // YYYY-MM-DD
    useGoal?: boolean;       // 是否用 goal:current 作为约束 (默认 true 如有 goal)
    inheritUnfinished?: {    // 跨月延续
      fromArchiveKey: string;     // 上月 archive key
      taskIds: string[];          // 用户勾选的未完成任务 id
    };
  }
  ```
- Response：SSE 流，每条事件：
  - `data: {"type":"thinking","text":"..."}` — thinking token (可选展示)
  - `data: {"type":"partial","plan":<部分Plan JSON>}` — 增量 plan
  - `data: {"type":"done","plan":<完整Plan>}`
  - `data: {"type":"error","message":"..."}`
- 用户审通过后另调 `PUT /api/plan` 保存

#### `POST /api/plan/parse-file`
- Content-Type：`multipart/form-data`
- Field：`file` (≤ 5MB)
- 支持类型：`.md` / `.txt` / `.pdf`
- 200 → `{ text: string, fileType: string, warnings?: string[] }`
- 400 → 不支持的文件类型 / 文件过大 / 解析失败
- 服务端：md/txt 直接 `text()`；pdf 用 `pdf-parse`，失败返 warnings 让用户知道

#### `GET /api/plan/onboarding-context`
- 月度重生成时前端调用，获取建议数据
- 200 →
  ```ts
  {
    hasGoal: boolean;
    goal: Goal | null;
    lastPlan: { archiveKey: string; title: string; endDate: string } | null;
    suggestedUnfinished: Array<{ taskId: string; content: string; reason: string }>; // 最多 5 条
  }
  ```
- 服务端跑一次轻量 AI 分析（non-thinking）识别未完成；缓存 1 小时

### 3.2 Snapshot

#### `GET /api/snapshot/[date]`
- Path param：`date` = `YYYY-MM-DD`
- 200 → `Snapshot`（不存在则从 baseline 生成并立即写入 + 返回）

#### `PUT /api/snapshot/[date]`
- Body：`Snapshot`
- 200 → `{ ok: true, snapshot: Snapshot }`

#### `POST /api/regenerate/[date]` (stream)
- Body：`{ thinking?: boolean }` (默认 true)
- Response：SSE
  - `data: {"type":"thinking","text":"..."}`
  - `data: {"type":"partial","snapshot":<部分>}`
  - `data: {"type":"done","snapshot":<完整>}`
- **不立即写入 KV**，前端拿到后让用户审，审通过后用户主动 PUT
- 服务端 prompt 自动拼装：plan.meta + 本周 + 前 N 天 snapshot + 昨天 reflection + 今天 baseline

### 3.3 Reflection

#### `GET /api/reflection/[date]`
- 200 → `Reflection` 或 `null`

#### `PUT /api/reflection/[date]`
- Body：`{ text: string }`
- 200 → `{ ok: true }`

### 3.4 Chat

#### `GET /api/chat/[date]`
- 200 → `ChatHistory`（不存在返 `{ messages: [] }`）

#### `POST /api/chat/[date]` (stream)
- Body：`{ message: string }`
- Response：SSE
  - `data: {"type":"token","text":"..."}` — 流式 token
  - `data: {"type":"tool_call","patch":[...]}` — AI 决定改 snapshot
  - `data: {"type":"done","message":<完整 ChatMessage>}`
- 服务端会同步：把 user message 写入 chat、把 assistant message 写入 chat、把 patch 应用到 snapshot

#### `DELETE /api/chat/[date]`
- 清空当天聊天历史
- 200 → `{ ok: true }`

- 200 → `Goal` 或 `null`

#### `PUT /api/goal`
- Body：`Goal` (Zod 校验)
- 200 → `{ ok: true, goal: Goal }`

## 4. 规范遵循声明（Standard Compliance Checklist）

### 4.1 命名

- 文件：kebab-case（`task-row.tsx`, `chat-drawer.tsx`）
- 组件：PascalCase（`TaskRow`, `ChatDrawer`）
- 函数 / 变量：camelCase
- 类型 / interface / Zod schema：PascalCase（`Plan`, `PlanSchema`）
- 常量（顶层 export）：UPPER_SNAKE_CASE（`KV_PREFIX`, `DEFAULT_MODEL`）
- API 路由：复数名词（`/api/snapshots/...` 不，单数 `/api/snapshot/[date]` 因为按 date 索引唯一）
- KV key：`{namespace}:{id}` 风格

### 4.2 格式化

- Prettier：默认配置 + `printWidth: 100`、`semi: true`、`singleQuote: false`
- ESLint：`eslint-config-next` + `@typescript-eslint/recommended-strict`
- TypeScript：`strict: true`、`noUncheckedIndexedAccess: true`、`exactOptionalPropertyTypes: true`

### 4.3 错误处理

- API route 顶层 try/catch，未捕获异常返 500 + 写日志
- Zod 校验失败 → 400 + `validation_error`
- KV 失败 → 500 + `kv_error` + 日志
- DeepSeek 失败 → 502 + `ai_error` + 错误细节（不含 key）
- 流式 endpoint 失败 → 发 `{type:"error"}` SSE 事件 + 关闭流
- 客户端：所有 mutation 用 TanStack Query `onError` 弹 toast
- **绝不 swallow error**：catch 后如果不处理就 re-throw

### 4.4 日志

- 服务端用 `console.log` JSON 格式（Vercel logs 兼容）：
  ```ts
  console.log(JSON.stringify({ level: "info", route, latencyMs, ... }))
  ```
- 不记录 PII（即使单用户）；不记录 API key；不记录完整 prompt（只记长度）
- AI 调用记：model、input tokens、output tokens、latency、是否 thinking

### 4.5 安全

- `DEEPSEEK_API_KEY` 仅服务端，client bundle 永不出现
- `APP_SECRET`（cookie token）服务端校验；首次访问跳 `/setup` 让用户输入 → 写 cookie
- `.env.local` 进 `.gitignore`，提供 `.env.example`
- `vercel.json` / `next.config.ts` 不含 secret
- 所有 AI 调用前对用户输入做 length 限制（< 50KB）防 DoS
- 如果未来开公开 URL，独立 endpoint 走只读 + 不带 token

### 4.6 测试

- 关键纯函数单元测试：`lib/date.ts`, `lib/schemas.ts` 边界、prompt builder
- KV roundtrip 集成测试：`tests/integration/kv.test.ts`（用 mock KV）
- 不写 e2e（v1 单人项目，ROI 低）

### 4.7 Git / Commit

- 分支：`main` 直接开发（单人项目，没必要 PR flow）
- Commit 规范：`<type>: <description>`，type ∈ {feat, fix, refactor, docs, chore, test}
- Commit message 用中文 OK
- 完成功能后**主动提醒 git push**（你 CLAUDE.md 要求）

### 4.8 部署

- 环境变量在 Vercel Dashboard 配置（不写进任何文件）
- `KV_REST_API_URL` / `KV_REST_API_TOKEN` / `DEEPSEEK_API_KEY` / `APP_SECRET` 四个
- preview 部署用同一 KV（单人项目无需隔离 prod/preview）
- Vercel project framework preset：Next.js（自动）

## 6. 风险 / 未知数

| 风险 | 应对 |
|---|---|
| DeepSeek thinking 模式可能不支持 function calling | M2 实现时验证；fallback：用 JSON mode + prompt 强约束输出 |
| AI 拆 baseline 输出不稳定（schema drift） | Zod 校验 + 失败时 retry；让用户能编辑 |
| Vercel KV 在自由档下命令配额（30K/月）超限 | 客户端缓存 + debounce；监控；超限只发生在重度使用 |
| 公开 repo + secret 泄漏 | pre-commit hook scan secret pattern；CI 加 gitleaks |
| 流式响应中断（网络断） | 客户端 detect SSE close 显示重试按钮 |
| AI 输出中文 JSON 字段名导致 Zod 报错 | prompt 显式要求英文 key；server 解析失败时 retry-once |
| 时间块冲突（两个任务同 5-7） | UI 上层叠显示 + 警告标记，不强制阻止 |
| iOS 加桌面引导差（用户找不到分享菜单） | 首次访问检测 iOS Safari + 未加桌面 → 弹卡片演示步骤；超过 1 次后不再提示 |
| 公开 repo 朋友 fork 后误用你的 KV 数据 | README 强调必须自己开 KV instance；`.env.example` 不放任何真实值 |
| 月度覆盖前用户操作失误（没 review 直接保存） | PUT /api/plan 时若已有 plan 必须传 `confirmOverwrite: true`，否则 409 |
| AI 输出 task 缺 tags/priority 字段 | Zod default 兜底（`tags: []`, `priority: "normal"`），不报错 |
| PDF 文本提取乱码 / 表格丢失 | parse-file 返 warnings 数组；强制让用户在 textarea 看见再交 AI 拆，不一气呵成 |
| "自动带未完成"被 AI 滥用堆任务 | 服务端硬限制 ≤ 5 条；prompt 明确要求"如果用户首周已满则不要再加" |
| Goal 文字过长导致 prompt 超 context | Goal 字段 max 5000 字（Zod 校验）；服务端拼 prompt 时如总长 > 限额，截断并日志告警 |
